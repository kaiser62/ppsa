# Phase 5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — hard architecture invariant forbids shadcn/any component library/build step (REQUIREMENTS.md "Architecture invariants", UI-03). Frontend stays plain JS/CSS/HTML at `docker/webui/app/static/index.html`, no CDN (offline/NetBird-only appliance). |
| Preset | not applicable |
| Component library | none — hand-rolled CSS components (`.card`, `.btn`, `.badge`, `.section`, `.modal`, `.alert`) defined once in a shared `<style>`/CSS block and reused verbatim across all 5 tabs in scope |
| Icon library | none — inline SVG only (currently the file uses raw emoji `🔓`/`🔒` for Wi-Fi security; replace with a tiny inline-SVG icon set — 4-6 glyphs total: lock/unlock, wifi-signal, check, warning, refresh, chevron — defined once as `<symbol>` in an SVG sprite at the top of `<body>` and referenced via `<use>`. No external icon font/CDN.) |
| Font | System font stack only (no font files/CDN, per offline constraint): `-apple-system, "Segoe UI", system-ui, sans-serif` for UI text, `"Cascadia Code", "SF Mono", Consolas, monospace` for the log/config/textarea `.config-editor` elements. This is a continuation of the current stack, not a change. |

**Source of pre-population:** the current `index.html` already has a `:root` CSS custom-property block (`--bg`, `--surface`, `--surface2`, `--primary`, `--primary-dark`, `--text`, `--text-muted`, `--green`, `--red`, `--yellow`) and a component-class system (`.card`, `.section`, `.btn`, `.badge`, `.alert`, `.bar-fill`, `.modal`). This UI-SPEC does not invent a system from scratch — it **tightens and extends** the existing one (adds spacing/typography tokens it currently lacks, fixes the flat/templated look) so the redesign reads as evolution, not replacement, and so existing state-handling JS (Phase 4's `server_state`/`players_known` logic) needs zero changes.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge internal padding |
| sm | 8px | Compact element spacing, form-row gaps on mobile |
| md | 16px | Default element spacing, card internal padding (reduced from current 20px `1.25rem`) |
| lg | 24px | Section padding, card grid gaps |
| xl | 32px | Header-to-content gap, page-level top margin |
| 2xl | 48px | Login card vertical centering breathing room |
| 3xl | 64px | not used at this container width (max-width 1200px) — declared for completeness only |

Exceptions:
- Touch targets: nav buttons and all `.btn`/`button` elements must have a minimum 44px tap height on viewports ≤768px (current `.nav button` padding of `0.4rem 0.75rem` ≈ 32px tall fails this on phone — bump mobile nav-button `padding` to `12px 16px` to clear 44px).
- Table cell padding stays at a tighter `8px 8px` (not on the 8-pt component scale exactly, but table density is an intentional exception for scanability — do not round up to 16px or rows become too tall on phone).

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 (regular) | 1.5 |
| Label | 13px | 400 (regular) | 1.4 — used for `.card h3`, `.form-group label`, table `th`, `.sub` micro-copy; uppercase tracking `0.04em` on table headers only |
| Heading | 20px | 600 (semibold) | 1.2 — used for `.section h2`, modal titles, nav-active state emphasis |
| Display | 28px | 600 (semibold) | 1.2 — used for dashboard stat `.card .value` numbers and the login-card `h1` (PPSA wordmark) |

Only 2 weights total (400, 600) — the current file's `header h1 { font-size: 1.25rem }` and ad-hoc inline `font-size` overrides (e.g. `style="font-size:1rem"` scattered through cards) must collapse into these 4 declared sizes; no more inline one-off `style="font-size:..."` per element.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0f172a` (existing `--bg`, slate-900) | Page background, body |
| Secondary (30%) | `#1e293b` (existing `--surface`, slate-800) for cards/sections/header/nav bar; `#334155` (existing `--surface2`, slate-700) for input borders, table row dividers, hover states, inactive nav pill background | Cards, section panels, header bar, table borders, form inputs |
| Accent (10%) | `#3b82f6` (existing `--primary`, blue-500), hover/active `#2563eb` (`--primary-dark`) | Reserved for: primary action buttons only (Sign In, Save & Apply, Save Changes, Connect, Save World, Backup Now, Save-File Backup), the active nav-tab underline/background, and focus rings on inputs. NOT used for status/semantic meaning (that's green/yellow/red below) and NOT used decoratively on headings, borders, or icons that aren't primary actions. |
| Destructive | `#ef4444` (existing `--red`) | Destructive actions only: Restart/Stop/Ban/Remove/Disconnect/Reset-to-defaults buttons (`.btn.danger`), error alerts, error status badges |

Additional semantic (status, not counted in the 60/30/10 — these are state indicators, not surface/accent colors):
- Success/healthy: `#22c55e` (existing `--green`) — running/connected/healthy badges, success alerts, low-usage bars
- Warning/degraded: `#eab308` (existing `--yellow`) — starting/inactive/degraded badges, warning alerts, mid-usage bars

Accent reserved for: Sign In button, Save/Apply/Connect/Save-World/Save-File-Backup/Backup-Now primary buttons, active nav tab indicator, input focus ring. Never applied to plain body links, card borders, or non-interactive decorative elements.

## Additional Design Notes (non-template sections, scoped context for planner/executor)

**Tabs in scope for this phase** (per hard constraints): Dashboard, Server Controls, Firewall, Backup, Wi-Fi. The other tabs visible in the current nav (Players, Config, System, Mods, WireGuard, NetBird, Settings) exist in the live file but are **out of scope for Phase 5's redesign pass** — do not restyle them as part of this phase's plan unless a shared component change (e.g. `.card`, `.btn`, `.alert`) unavoidably touches them too. If the shared CSS system change does cascade to out-of-scope tabs, that's acceptable (that's the point of a shared system, UI-02) — just don't spend plan budget hand-polishing their unique markup.

**What "non-templated" means here (diagnosis of current state):** the existing UI isn't broken, it's *generic* — Tailwind-esque utility-flavored inline styles (`style="font-size:1rem"`, `style="margin-top:0.5rem"` scattered ~15+ times), inconsistent spacing (`0.5rem`/`0.75rem`/`1rem`/`1.25rem` all used interchangeably with no scale), a flat single-elevation surface (every card is the same `--surface` slate with no depth/hierarchy cue between primary content and secondary chrome), and no distinguishing brand personality beyond the login screen's "PPSA" wordmark. The redesign's job is: (1) collapse inline `style=` spacing into the 8-pt scale via classes, (2) add one subtle elevation cue (e.g. 1px border in `--surface2` or a soft shadow) to distinguish `.card`/`.section` from body background rather than relying on color-fill alone, (3) apply the 4-size type scale consistently instead of ad-hoc inline `font-size`, (4) carry the login screen's brand moment (PPSA wordmark treatment) into the header so the app doesn't feel like a different product than its own login page.

**Responsive breakpoint:** single breakpoint at 768px is sufficient (laptop vs phone, per UI-04's two named targets) — do not over-engineer a 3-4 breakpoint system for a 2-device requirement.

**Dark-only:** confirmed no light/dark toggle in this phase (UI-V2-02 is explicitly deferred to v2 in REQUIREMENTS.md) — the existing dark slate palette is the permanent aesthetic, not a placeholder.
