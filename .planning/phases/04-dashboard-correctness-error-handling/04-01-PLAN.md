<objective>
Make the WebUI dashboard tell the truth about the Palworld server: show the real game
version reliably (even after the startup log line rotates out), distinguish "still
starting" from "empty" from "broken" on a fresh boot, render explicit empty states
instead of misleading zeros/blanks, and degrade gracefully with an actionable banner
when an upstream/status call fails or times out.

Context — this phase HARDENS already-committed code. A prior ad-hoc session committed
`4a415dc fix(webui): Phase 4 dashboard correctness (DASH-01..05)`, which added `server_state`,
a `--tail 50` log-parse version fallback, and a state banner. That commit is a good baseline but
has verified correctness gaps this plan closes:
  - DASH-01: version reverts to blank once the one-time `Game version is vX` startup log line
    scrolls past the 50-line tail window on a long-running server (no last-known-good cache).
  - DASH-03: the frontend still shows a misleading `0` player count during `starting`/`unavailable`.
  - DASH-04: `dashboard()` awaits three upstream calls serially at 5s each, so a dead upstream
    hangs the handler up to 15s; and `/api/system` still raises a bare 500 on any probe error.
Executor: treat the existing code as the baseline to READ FIRST and modify in place, not a blank slate.

Purpose: Deliver honest, non-misleading dashboard data so the Phase 5 visual redesign dresses
correct states rather than the buggy current output.
Output: Hardened `dashboard()` + `system_health()` in main.py, honest state-aware rendering in
index.html, and a verification checkpoint confirming all five DASH requirements on real behavior.
</objective>

    Modify dashboard() to fix three verified backend gaps; build on the existing server_state and
    version-fallback logic rather than rewriting the whole handler.

    (a) DASH-01 version durability — the "Game version is vX" line is printed once at Palworld startup
    and scrolls out of the current --tail 50 window on a long-running server, so version reverts to
    unavailable. Fix with a module-level last-known-good cache: introduce a module-level mutable holder
    (name it _VERSION_CACHE, a dict with a single key "value" initialized to empty string) near the other
    module constants. In dashboard(), resolution order for version becomes: (1) non-empty REST info.version;
    else (2) parse the container log for the Game-version pattern via a case-insensitive regex — widen the
    log window from 50 to a larger tail (use 400) so a recently-booted server still has the line; else
    (3) the cached _VERSION_CACHE value if previously resolved. Whenever (1) or (2) yields a real version,
    write it back into _VERSION_CACHE. Only emit the literal string unavailable when all three are empty.
    Add a ponytail comment explaining the log-rotation reason for the cache.

    (b) DASH-04 no-hang — the handler currently awaits palworld_get for /info, /metrics, /players serially,
    each with the default 5s timeout, so a dead upstream blocks the handler up to 15s. Replace the three
    serial awaits with a single concurrent asyncio.gather over those three coroutines using
    return_exceptions=True, so total upstream wait is bounded by one timeout rather than three. Derive
    rest_ok from whether the /info result is a real dict (not an Exception instance). For any call that
    returned an Exception, substitute the same safe defaults already used (info as an error dict, metrics as
    an empty dict, players as an empty list). Keep server_state derivation
    (running/starting/stopped/unavailable) behaviorally identical.

    (c) DASH-03 honest counts — add a signal the frontend can trust: keep player_count but ALSO return
    players_known (true only when rest_ok and players is a list) so the frontend can distinguish
    "server up, genuinely 0 players" from "server not up yet, count unknown". Do not remove any existing
    response key; server, metrics, players, player_count, server_state, version all remain.

    Then modify system_health() for DASH-04 graceful degradation: it currently wraps everything in one
    try/except and raises HTTPException 500 on any failure, which breaks the System view entirely on a
    transient probe error. Change it to degrade per-section: on failure of an individual probe (meminfo,
    loadavg, disk, uptime, docker ps) substitute an empty/null value for that section and continue, and
    return HTTP 200 with a top-level degraded boolean set true plus a short detail string naming what
    failed, instead of a bare 500. The endpoint must still return 200 with a usable (possibly partial)
    payload rather than raising.
  </action>
