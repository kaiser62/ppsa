<objective>
Extend `scripts/ppsa-installer-e2e.py` (Phase 6/7 deliverable) to invoke the
existing `scripts/ppsa-smoke-test.py` as a final pipeline stage via
`subprocess.run()`, fold its pass/fail result into the same ordered
`results` list / `overall_pass` computation Phase 7 established, write all
raw smoke-test subprocess output to a fixed (overridable) log file, and
produce a one-line human-readable summary that names which stage failed on
an overall FAIL verdict (TEST-01, TEST-02).

Purpose: This is the final phase of the v1.5.0 milestone — after this plan,
a tester runs one script and gets one verdict covering install, boot
verification, and functional smoke test, with all raw noise routed to a
log file instead of the main console.

Output: `scripts/ppsa-installer-e2e.py` updated in place with a new
`run_smoke_test()` method, two new CLI flags (`--skip-smoke-test`,
`--log-file`), and a one-line summary builder wired into `run()`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ppsa-installer-e2e.py -> ppsa-smoke-test.py subprocess | Local subprocess invocation of a sibling script in the same repo; not a network/remote trust boundary, but still an external process whose stdout/stderr/exit-code this script consumes and trusts |
| ppsa-installer-e2e.py -> filesystem (log file) | Writes captured subprocess output to a log file path that may be operator-supplied via `--log-file` |
| ppsa-smoke-test.py -> target VM (SSH) | Existing trust boundary, unchanged by this phase -- smoke test itself owns its SSH transport and credentials |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-08-01 | Tampering | `--log-file` operator-supplied path | low | accept | Path is a local CLI argument supplied by the same operator running the tool (not untrusted network input); opened in `"w"` mode which can overwrite an existing file at that path -- acceptable for a disposable manual test-tool, consistent with D-02's explicit "no retention policy needed" decision. No path traversal risk beyond what the operator's own shell access already grants. |
| T-08-02 | Denial of Service | `run_smoke_test()` subprocess invocation | medium | mitigate | `subprocess.run(..., timeout=600)` bounds the smoke-test subprocess call so a hung smoke test cannot hang the whole e2e pipeline indefinitely; `subprocess.TimeoutExpired` is caught and mapped to a `FAIL` result rather than propagating an uncaught exception. |
| T-08-03 | Information Disclosure | Raw smoke-test subprocess output (may contain JWT tokens, SSH command text) written to `self.log_file` | low | accept | This matches the existing behavior already accepted for `ppsa-smoke-test.py`'s own `--log-dir` raw output (same repo, same threat class, already in production use per Phase 2's precedent) -- the log file is local-disk-only, never transmitted, and the operator running this tool already has SSH credentials to the target VM in-hand. |
| T-08-04 | Repudiation | Smoke-test subprocess invocation not independently auditable beyond this script's own log file | low | accept | Single-operator, manual, disposable-VM test tool -- no multi-user audit requirement exists for this tool class (consistent with Phase 6/7's threat registers, which accepted equivalent local-invocation risks). |
| T-08-05 | Elevation of Privilege | `[sys.executable, str(SMOKE_TEST_SCRIPT), ...]` subprocess argument construction | low | accept | `SMOKE_TEST_SCRIPT` is a fixed, hardcoded sibling-file path (not operator-controllable, per D-03's explicit "no flag for the smoke-test script's own path" decision) -- no argument injection surface from user input into the script path itself; `ssh_target` and `ssh_password` are passed as separate argv list elements (not shell-interpolated), so `subprocess.run()`'s list-form invocation (no `shell=True`) prevents shell metacharacter injection from a malicious `--ssh-target`/`--ssh-password` value. |

</threat_model>
