<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> POST /api/backup/restore-upload | User selects a .tar.gz file from their local filesystem; the browser reads it via File API and sends to the webui backend (backend validates structure per Plan 01's _validate_save_archive) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-FE-01 | Spoofing | Restore confirmation modal | low | mitigate | Restore always requires explicit user confirmation via confirmAction() before calling the API; the modal message explains the destructive nature (stops Palworld, restores save) so the user cannot accidentally trigger restore |
| T-03-FE-02 | Repudiation | Save-file backup trigger | low | accept | Consistent with the existing triggerBackup() pattern — no separate confirmation needed; the action is non-destructive |
| T-03-FE-03 | Tampering | Upload file type filter | low | mitigate | The `.tar.gz` accept attribute on the file input is a client-side hint; backend _validate_save_archive provides the real gate. Client-side accept is defense-in-depth only |
</threat_model>
