<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| test-host -> test VM (SSH) | Script SSHs into a freshly-installed, untrusted-until-verified guest using hardcoded default credentials (`ppsa`/`ppsa`) |
| test-host -> VBoxManage subprocess (keyboardputscancode) | Script injects raw keystrokes into a running VM; scancode content is a fixed constant, not attacker-controlled input |
| CLI --ssh-target -> subprocess/SSH argv | User-supplied address flows into plink/ssh command construction |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-06-05 | Spoofing | wait_for_install_complete SSH host-key handling | medium | mitigate | Reuse ppsa-smoke-test.py's accept-on-first-connect pattern (StrictHostKeyChecking=accept-new / plink -batch with prompted key acceptance) — acceptable for a disposable, freshly-created test VM whose identity is not yet known, consistent with existing smoke-test precedent |
| T-06-06 | Information Disclosure | Default SSH credentials (ppsa/ppsa) hardcoded | low | accept | These are the appliance's own documented first-boot defaults (not a secret), identical to the existing ppsa-smoke-test.py precedent; test VMs are disposable and never carry production data |
| T-06-07 | Denial of Service | wait_for_install_complete polling loop | medium | mitigate | Bounded overall_timeout_seconds enforced; loop always terminates and reports a distinguishable failure reason rather than blocking indefinitely |
| T-06-08 | Tampering | send_scancodes keyboardputscancode argv | low | accept | Scancode bytes are a fixed module-level constant (INSTALLER_TUI_SEQUENCE), never derived from CLI/network input — no injection surface |

</threat_model>
