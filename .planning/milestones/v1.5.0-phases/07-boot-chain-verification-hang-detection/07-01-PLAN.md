<objective>
Add a minimal, additive heartbeat mechanism to `scripts/install.sh`'s first-boot flow: a
`mark_step_activity()` helper (matching the existing `mark_step()` pattern) that writes a
Unix timestamp to `/run/ppsa-install.activity`, wired into Step 3 (Deploying Docker stack)
so a poller outside the guest can distinguish "still pulling/starting containers, just
slow" from "genuinely stalled" — this is the guest-side half of BOOT-02 (hang detection).

Purpose: `scripts/ppsa-installer-e2e.py` (extended in Plan 07-02) needs a heartbeat signal
to poll over SSH during the ~10-20+ minute Docker-pull window, so its hang-detection logic
has real signal to read instead of guessing from a single fixed timeout.

Output: `scripts/install.sh` with an additive `mark_step_activity()` helper and heartbeat
writes inside Step 3's docker pull/up loops, with zero changes to the existing
`mark_step()` step-counter contract that `ppsa-firstboot.sh`'s tty1 display and
`/run/ppsa-install.progress` depend on.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| first-boot guest (install.sh) -> host test harness (SSH poll) | The heartbeat file is written by root-run install.sh inside the guest and read by an unprivileged `ppsa` SSH session from the host-side tester |
| /run tmpfs -> world-readable heartbeat file | The heartbeat timestamp file must be readable without sudo, widening its readability beyond root-only defaults |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-07-01 | Information Disclosure | /run/ppsa-install.activity world-readable (644) | low | accept | The file contains only a Unix timestamp (no secrets, no config, no credentials) — readability is required for the SSH-polling hang-detection mechanism to function at all; equivalent exposure already exists for /run/ppsa-install.progress |
| T-07-02 | Denial of Service | mark_step_activity() write failures under set -eu | low | mitigate | Both the timestamp write and the chmod use `2>/dev/null \|\| true`, matching the existing mark_step()/LOG_FILE pattern, so a /run write failure (e.g. read-only tmpfs edge case) can never abort first-boot install under `set -eu` |
| T-07-03 | Tampering | Heartbeat file overwritten by a malicious local process during install | low | accept | Install.sh runs as root during first-boot on a freshly-imaged, single-tenant appliance with no untrusted local processes present at that stage; this is a local test-harness signal, not a security control |

</threat_model>
