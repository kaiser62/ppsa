<objective>
Replace the blind-console/ufw-exception SSH bootstrap in the current installer test
recipe with a persistent, low-token NetBird-overlay SSH path: a freshly installed test
VM gets a dedicated, non-production NetBird identity, a fixed hostname that yields a
stable NetBird DNS label, and reaches SSH through the firewall rule that already exists
today — no appliance code change, no new iptables/ufw rule, no shared identity reuse.

Purpose: Phase 2's scripted smoke test needs a stable, reproducible SSH endpoint to run
against. This phase proves that endpoint exists and is repeatable without re-deriving
the mechanism each session.

Output: A local-only test-key config convention (mirroring the existing
`netbird.local.json` pattern), a live re-enrollment procedure that assigns the test VM
a fixed hostname and isolated NetBird identity post-boot, a verified SSH-over-NetBird
connection to a real CI-built test VM, and updated docs/skill capturing the procedure.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dev-peer -> test-VM over NetBird overlay | SSH traffic from an authenticated NetBird peer into the test VM's sshd |
| local filesystem -> git repo | netbird.test.json (secret-bearing) sits beside tracked files; risk is accidental `git add -f` or gitignore regression |
| console injection -> test VM tty1 | blind command injection into an auto-logged-in root-capable shell (pre-existing sanctioned mechanism, not introduced by this phase) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-01-01 | Information Disclosure | netbird.test.json / netbird.test.json.example | high | mitigate | .gitignore entry added in Task 1 before the file is ever created locally; Task 1 acceptance criteria greps for accidental secret-shaped strings in tracked files; checkpoint after Task 2 explicitly re-verifies via `git log -p` that the real key never entered history |
| T-01-02 | Spoofing / Identity collision | Test VM hostname + NetBird peer identity | medium | mitigate | Fixed hostname `ppsa-test` is applied only post-boot to a VM already using its own unique NetBird peer identity (never reuses the appliance's PPSA_NB_SETUP_KEY or a shared IP like the WG 10.8.0.2 precedent); Task 1 action explicitly notes not to run two `ppsa-test`-named VMs concurrently |
| T-01-03 | Elevation of Privilege | Console-injected commands during re-enrollment (Task 2) | low | accept | Console injection into tty1 is an already-sanctioned, pre-existing access path (ppsa-guest-ops SKILL.md) for a test-only VM the operator physically controls in VirtualBox; no new privilege boundary is crossed by this phase |
| T-01-04 | Tampering | WG_FRIENDS firewall chain | low | accept | This phase deliberately makes zero firewall/iptables/ufw changes (D-03); Task 2 acceptance criteria positively verifies no `ufw` command was run and the chain is read-only inspected |
</threat_model>
