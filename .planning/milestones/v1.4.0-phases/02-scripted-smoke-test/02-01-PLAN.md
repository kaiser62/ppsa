    --- Architecture ---

    The script has three layers:

    1. **Transport layer** (`_ssh_exec`) — runs a command on the remote VM and returns
       (stdout, stderr, exit_code). On Windows, uses `plink -ssh -batch -hostkey ... -pw
       ... ppsa@<host> <cmd>`. On Linux/macOS, uses `ssh -o BatchMode=no -o
       StrictHostKeyChecking=accept-new ppsa@<host> <cmd>`. The host key is auto-accepted
       on first connection. SSH credentials: user `ppsa`, password `ppsa` (default;
       overridable via CLI flags). Returns decoded UTF-8 text.

    2. **Check layer** (`run_check`) — a function that accepts a check name, an SSH
       command string, and one or more assertions. The function runs the command via
       `_ssh_exec`, applies the assertions, records PASS/FAIL, and appends the raw
       command+output to the log file. Supported assertions:
       - `exit_zero`: exit code must be 0
       - `json_has(path, value)`: parse stdout as JSON and assert a nested key equals
         value (e.g. `json_has("status", "ok")`, `json_has("chain_present", true)`)
       - `json_has_key(path)`: assert a JSON key exists (any value)
       - `matches(regex)`: stdout must match a regex pattern
       - `count_lines(op, n)`: stdout line count must satisfy op (eq/gt/ge/lt/le) vs n
       - `response_time(max_seconds)`: command must complete within N seconds
       - Multiple assertions per check are ANDed; the first failure marks the check FAIL.
       Each check returns a `CheckResult` tuple: (name, passed: bool, detail: str).

    3. **Runner layer** — iterates over the check definitions grouped by category.
       Before checks, writes a header to the log file with timestamp, target, and
       command line. After all checks, prints the summary table and returns exit code 0
       (all pass) or 1 (any fail).

    --- Plink host-key handling ---

    `plink` prompts interactively for host-key acceptance on first connection, which
    breaks `-batch` mode. The script must handle this:
    1. On first connection to a VM, run plink in non-batch mode once to accept the host
       key: `echo y | plink -ssh ppsa@<host> "hostname -s"`. Capture the SHA256
       fingerprint from the output.
    2. On subsequent connections (all check commands), use `-batch -hostkey
       "SHA256:<fingerprint>"` to avoid the prompt.
    3. Store the host key in memory for the duration of the run (no persistence needed
       between runs — test VMs get rebuilt).
    On Linux/macOS, use `StrictHostKeyChecking=accept-new` which auto-accepts on first
    connection.

    --- Error handling ---

    - If SSH connection fails entirely (wrong address, VM not booted): print a clear
      error, write to log, exit 2 (connection error).
    - If any single check assertion fails: mark that check FAIL, continue to next check.
    - If all checks pass: exit 0.
    - If any check fails: after all checks complete, print failed checks list and exit 1.
    - The script should catch `subprocess.TimeoutExpired` on individual checks and mark
      them FAIL (with detail "timed out after N seconds") rather than crashing.
    - Interrupt (SIGINT/SIGTERM) should flush the log file before exiting.

    --- What NOT to do ---

    - Do NOT use an SSH key by default (the VM uses password `ppsa` for the `ppsa` user)
    - Do NOT require `paramiko` or any pip packages — the script must work with stdlib
      only (subprocess, json, sys, os, time, datetime, argparse, platform, pathlib)
    - Do NOT assume the test VM has a specific hostname or NetBird DNS label — take it
      as a positional argument
    - Do NOT write to the VM's filesystem (no pscp, no file copy) — all interaction is
      SSH command execution
    - Do NOT check ppsa-wgdashboard container unconditionally — it only runs when WG is
      enabled (non-default)
    - Do NOT hardcode {{.Names}} docker format strings that use Go template syntax
      inside SSH-quoted strings without proper escaping — double-check the quoting works
      through the SSH layer
