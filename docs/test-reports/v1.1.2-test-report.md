## 🎯 Feature: tty1 First-Boot Progress UI

### What was added (v1.1.1 → v1.1.2)

- **`ppsa-firstboot.sh`** — a bash script that owns /dev/tty1 during first boot
  and renders a live progress UI: PPSA version banner, progress bar, 7-step
  status list with DONE/ACTIVE/WAITING markers, live status line from the
  install log, and a completion screen with all connection URLs.
- **`ppsa-firstboot.service`** — systemd unit that runs the script in
  foreground on /dev/tty1 (`StandardInput=tty`, `StandardOutput=tty`,
  `TTYPath=/dev/tty1`). Uses `Conflicts=getty@tty1.service` so the getty
  yields the tty until firstboot finishes. Gated by
  `ConditionPathExists=!/opt/ppsa/.installed` so it only runs once.
- **`install.sh`** — added a `mark_step` helper that writes the current
  step to `/run/ppsa-install.progress` as the install progresses. Renumbered
  steps from 0-6 to 1-7 (7 total) so progress math is clean.
- **getty@tty1.service.d/autologin.conf** — overrides the tty1 getty with
  `--autologin ppsa`, so after the firstboot UI exits, the user is dropped
  to a shell without a password prompt.
- **/etc/profile.d/ppsa-welcome.sh** — shows the WebUI/SSH URLs on every
  interactive login (SSH or local shell).
- **VERSION file** — build script writes `/opt/ppsa/VERSION` with the
  latest git tag (or a date stamp in dev builds), used by the tty1 banner.

### 🐛 Bug Found & Fixed During Testing

**Bug:** `ConditionPathExists` and `Conflicts` were placed in `[Service]`
section. These directives belong in `[Unit]`. systemd silently ignored them:

```
/etc/systemd/system/ppsa-firstboot.service:16: Unknown key
    'ConditionPathExists' in section [Service], ignoring.
/etc/systemd/system/ppsa-firstboot.service:17: Unknown key
    'Conflicts' in section [Service], ignoring.
```

**Fix:** Moved both directives to `[Unit]`. Verified in v1.1.2 build — journal
no longer shows the warnings, and the service correctly:
- Skips running on subsequent boots (ConditionPathExists)
- Stops the tty1 getty while it runs (Conflicts)

The first v1.1.2 build had this bug; the fix was committed (f03cf19) and
re-tagged. The download on GitHub release v1.1.2 is the fixed build.
