## ­¤Ä» What Was Added in v1.1.6

The single change shipped in v1.1.6:

- **`compose/docker-compose.yml:89`** ŌĆö added
  `- /etc/ppsa:/etc/ppsa:rw  # ponytail: override /:/host:ro for this dir so iptables-save persists across reboots. Docker's most-specific-path resolution makes this rw.`
  to the `ppsa-webui` service. The intent was that when the webui container runs
  `ppsa-firewall-apply.sh` and that script calls `iptables-save > /etc/iptables/rules.v4`
  (which is under the `/:/host:ro` parent mount), the new `/etc/ppsa:/etc/ppsa:rw`
  mount would be the most-specific path resolution and allow writing to that
  subdirectory.

### What the rw mount fix actually did

The new `/etc/ppsa:/etc/ppsa:rw` mount in `docker-compose.yml:89` IS working
correctly. Verified by writing a test file from inside the webui container:

```
$ docker exec ppsa-webui sh -c "echo test > /etc/ppsa/test-write.txt && cat /etc/ppsa/test-write.txt"
test
WRITE OK
```

The mount works, the directory is on the host's root filesystem
(`/dev/sda2`, 12G, persistent), and files written from the webui container
are visible on the host after reboot. **But the firewall apply script does
not use this path.** It still tries to write to `/etc/iptables/rules.v4` (under
the read-only `/:/host:ro` mount) and falls back to
`/var/lib/docker/volumes/compose_webui_data/_data/iptables.rules.v4`, which
the webui container sees as read-only because:

- The `webui_data` Docker volume is mounted at `/app/data` in the container
  (writable).
- The script uses the host's path `/var/lib/docker/volumes/compose_webui_data/_data`
  which is reachable as `/host/var/lib/docker/volumes/...` (read-only) inside
  the container.

So the rw mount fix on `/etc/ppsa` is a necessary building block but it is
**not sufficient**. To make rules survive reboot, two more pieces are
needed:

1. **`ppsa-firewall-apply.sh` should write the iptables-save output to
   `/etc/ppsa/iptables.rules.v4`** (not `/etc/iptables/rules.v4`). This is
   the path the new rw mount actually unblocks.
2. **A systemd service (e.g. `ppsa-firewall-restore.service`) that runs
   `iptables-restore < /etc/ppsa/iptables.rules.v4` on boot** ŌĆö or simply
   re-runs `ppsa-firewall-apply.sh` (which is idempotent and reads the
   config from the persistent webui_data volume).

## ­¤Éø Issues Found

### 1. The rw mount fix is incomplete (high priority)

The `/etc/ppsa:/etc/ppsa:rw` mount in `docker-compose.yml:89` is correct but
`ppsa-firewall-apply.sh` does not use it. The script's `RULES_V4` is still
`/etc/iptables/rules.v4` and the fallback `WEBDATA` is still the host's
`/var/lib/docker/...` path. Neither path is writable from inside the
webui container, so:
- iptables-save output is never persisted to a file from the webui.
- The chain lives only in kernel memory and is lost on reboot.
- The user-visible effect: WebUI firewall changes do not survive reboot.

**Workaround that proves the fix path:** manually run
`ppsa-firewall-apply.sh` on the host (not from the webui) and the chain is
rebuilt from the persistent `webui_data/firewall.json`. The host version of
the script CAN write to `/etc/iptables/rules.v4` (it is the host, no
`/:/host:ro` mount), but even that file is not loaded on boot because
`netfilter-persistent` is not installed and there is no service that does
`iptables-restore` on boot.

### 2. WebUI status endpoint always reports `chain_present: true`

`docker/webui/app/main.py:943`:
```python
return {"rules": out, "chain_present": "WG_FRIENDS" in out}
```

The status endpoint runs `iptables-save 2>/dev/null | grep ... || echo
'(WG_FRIENDS chain not configured)'`. The literal string "WG_FRIENDS" appears
in BOTH the populated case (real rules) AND the fallback "(WG_FRIENDS chain
not configured)" message. So `chain_present` is always `true`. The frontend
status badge (index.html:907) therefore always shows "Chain is active on the
host." even when the chain has been wiped by a reboot.

**Fix:** check `rc == 0` from the host exec, or test for an actual `-A
WG_FRIENDS` rule in the output, or compare to a sentinel value that
excludes the fallback message.

### 3. `wg-quick` PATH requires `/sbin` (low priority, not a v1.1.6 regression)

`ppsa-wireguard-register.sh` invokes `wg-quick` which calls
`resolvconf` from wg-quick line 32. `resolvconf` is in `/sbin` and
`/usr/sbin`, not in `ppsa`'s default `PATH` (the `openresolv` package IS
installed; the binary is just outside the default user PATH). Worked around
by setting `PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin`
before running the script. Same issue would happen on v1.1.5.

**Fix:** add `/sbin` to `/etc/environment` or to the user's `.bashrc` in the
image build, or invoke wg-quick with an absolute path in the registration
script.

### 4. `apt-get install -y -qq resolvconf` returned silently with no effect

The first attempt to install `resolvconf` via `apt-get install -y -qq` did
not surface any error but the binary was still missing. It turned out
`openresolv` was already installed, and what the script was missing was
just being on `PATH`. Cosmetic, but a louder failure mode would have
saved a few minutes.

## ­¤¤ó RELEASE VERDICT: v1.1.6 is NOT READY for the persistence story

The headline change in v1.1.6 ŌĆö the `/etc/ppsa:/etc/ppsa:rw` mount fix ŌĆö is
a correct and necessary step toward making firewall rules survive reboot,
but **on its own it does not deliver persistence**. The user-visible bug
("I change the firewall in the WebUI and the changes vanish on reboot") is
not fixed by v1.1.6.

What works in v1.1.6:
- Ō£ģ First-boot install (8/8, including the Wi-Fi-onboard fix from earlier)
- Ō£ģ Docker stack
- Ō£ģ WireGuard auto-registration (auth, peer create, config download, wg0 up,
  handshake, systemd enable, survives reboot)
- Ō£ģ Firewall WebUI (config GET/PUT, apply, status)
- Ō£ģ WebUI config persists across reboot (via webui_data Docker volume)
- Ō£ģ `wg0` interface comes back up across reboot with the same IP

What is broken in v1.1.6:
- ŌØī Firewall rules in the kernel do not survive reboot
- ŌØī WebUI status badge always says "Chain is active" (false positive)
- ŌÜĀ’ĖÅ resolvconf PATH for the WG registration script (pre-existing)

### Recommended fixes for v1.1.7

1. Update `ppsa-firewall-apply.sh`:
   - Change `RULES_V4="/etc/iptables/rules.v4"` to
     `RULES_V4="/etc/ppsa/iptables.rules.v4"`.
   - Change `WEBDATA="/var/lib/docker/volumes/compose_webui_data/_data"` to
     `WEBDATA="/app/data"` (the in-container mount point, writable).
2. Add `scripts/ppsa-firewall-restore.service` + `.sh`:
   - `Type=oneshot`, `RemainAfterExit=yes`.
   - `[Service] ExecStart=/opt/ppsa/scripts/ppsa-firewall-apply.sh`.
   - `[Install] WantedBy=multi-user.target`.
   - Installed by `install.sh` alongside the wifi-onboard and install services.
3. Fix `docker/webui/app/main.py:943` so `chain_present` only returns `true`
   when the chain is actually configured (e.g. parse the rc of the host
   exec, or check for `-A WG_FRIENDS` in the iptables-save output).
4. Add `/sbin` to the `ppsa` user's `PATH` in the image build
   (`/etc/environment` or `/etc/profile.d/`).
